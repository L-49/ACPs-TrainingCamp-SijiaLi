# LLM辅助开发过程记录

## 1. 开发工具和使用场景

在本次项目开发过程中，我使用了通义千问（Qwen）作为**辅助参考工具**，主要用于以下场景：
- 查阅技术文档和最佳实践
- 验证特定技术方案的可行性  
- 调试复杂问题时获取思路启发

## 2. 具体使用过程记录

### 2.1 DSL语法格式选择
在项目初期，我对DSL的语法格式（YAML vs JSON vs 自定义格式）存在疑问。通过查询通义千问，了解到YAML格式具有良好的可读性和注释支持，适合配置类文件。基于这一参考信息，结合项目需求，我设计了当前的YAML DSL语法结构。

### 2.2 技术难点调试
在开发过程中遇到两个主要技术难点：

**难点1：占位符替换机制**
- 问题：如何在DSL响应模板中动态插入用户输入内容
- 参考过程：查询通义千问关于"Python字符串模板替换"的最佳实践
- 独立实现：基于参考信息，我设计并实现了`{{input}}`占位符替换机制，在`execute_actions`方法中完成具体编码

**难点2：意图识别优化**  
- 问题：4位数字订单号等特殊输入的意图识别准确率不高
- 参考过程：咨询通义千问关于"LLM提示词工程"的建议
- 独立实现：根据提示词优化原则，重新设计了意图识别的prompt模板，并添加了关键词回退机制

### 2.3 代码质量验证
在完成各模块开发后，我使用通义千问对关键代码段进行**安全性检查**和**性能评估**，确保代码符合Python最佳实践。所有改进建议都经过我的仔细评估和手动修改。

## 3. 使用频次和程度说明

- **总使用次数**：约5-6次（主要集中在技术调研和问题调试阶段）
- **使用方式**：以问答形式获取技术参考信息

## 4. 个人收获和反思

通过本次项目，我深刻体会到：
1. **独立思考的重要性**：即使有AI工具辅助，核心的设计决策和问题解决仍需依靠自己的技术积累
2. **批判性思维的必要性**：AI提供的建议需要经过严格验证和适配，不能盲目采纳，很多内容可能与客观事实不符
3. **工程实践的价值**：从需求分析到测试验证的完整开发流程，锻炼了我解决实际工程问题的能力
